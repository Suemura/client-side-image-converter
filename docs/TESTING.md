# テスト方針の詳細

機能別の単体テスト / E2E の対象一覧。原則は CLAUDE.md の「テスト方針」を参照。

- テストランナーは vitest、DOM 環境は happy-dom（`File` / `FileReader` / `localStorage` などのブラウザ API を使用するため）
- テストファイルはテスト対象と同じディレクトリの `__tests__/` に `<対象ファイル名>.test.ts` として配置する
- `src/utils/` のロジックを追加・変更した場合は、対応する単体テストを追加・更新すること
- EXIF 処理のテストは piexifjs でフィクスチャ（EXIF 入り JPEG）を生成して実データで検証する（`metadataManager.test.ts` / `exifTransfer.test.ts` 参照。piexifjs の動的 import 化（Issue #114）に伴い `readExifTiffFromDataUrl` / `normalizeExifForBakedImage` 等の呼び出しは await する）
- C2PA バイナリ処理（Issue #104）は、JUMBF の検出・解析（ISO 19566-5）・ロスレス除去の純粋ロジックを単体テスト（`c2paBinary.test.ts` 13 件: JPEG APP11 の単一セグメント / マルチセグメント / C2PA 無し、PNG caBX の有無、WebP C2PA チャンクの有無、除去後の形式検証等）・マニフェストストア JSON の整形（`c2paSummary.test.ts` 9 件: 署名状態・AI 生成判定・編集履歴のパース・防御的部分型・欠損フィールドの fail-closed）・メタデータ管理への検出ゲート（`metadataManager.test.ts` に +4 件: `detectC2pa` の JPEG/PNG/WebP 判定）し、実ブラウザの署名検証（`c2paManager.ts` の `@contentauth/c2pa-web` WASM ロード・verifyTrust 無効のクリプト検証）と metadata ページの UI（`ProvenanceSection.tsx` の来歴表示・除去チェックボックス）・E2E の送出（`e2e/metadata.spec.ts` に +4 件: C2PA 検出時の来歴表示・AI 生成バッジ・署名状態の確認・除去チェック後の出力検証）で検証する
- Canvas API や WASM / WebGL に依存する処理（描画・変換・エンコード）は happy-dom では動作しないため、単体テストの対象外とする（純粋ロジック部分を切り出してテストする）。編集ページも同様で、色調整の数式（`adjustments.ts` の `applyAdjustmentToPixel`（ガンマ / モノクロ含む）等）・ディテール / ビネット / グレインの数式（`effects.ts` の `detailDeltaAt` / `vignetteFactorAt` / `lowbias32` / `grainNoiseAt` 等。ガウスカーネルの対称性・ハッシュの決定性と乗数のビット同一ガードを含む）・シェーダ生成/配線（`adjustmentShader.ts`）・LUT のパースと CPU 適用（`lutParser.ts` の `parseCubeLut` / `applyLutToPixel` / `haldClutToLutData` 等）・LUT の選択解決（`lutState.ts` の `resolveLutForIndex` 等）・トーンカーブの点操作・補間・テーブル焼成と CPU 適用（`toneCurve.ts` の `addCurvePoint` / `evaluateCurve` / `buildToneCurveTable` / `applyToneCurveToPixel` 等）・ヒストグラムのビン計算と SVG パス生成（`histogram.ts` の `computeHistogram` / `resolveHistogramSampleSize` / `buildHistogramPath` 等）・自動補正 / WB スポイトのスライダー値逆算とサンプリング（`autoAdjust.ts` の `histogramPercentileRange` / `computeAutoLevels` / `computeWhiteBalanceForNeutralPoint` / `computeAutoWhiteBalance` / `clampSampleWindow` / `averageRgb` / `displayPointToSourcePixel` 等。逆算値を `applyAdjustmentToPixel` へ通すクロスチェック・gray-world との等価性回帰を含む）を純粋ロジックとして単体テスト（`adjustments.test.ts` / `effects.test.ts` / `adjustmentShader.test.ts` / `lutParser.test.ts` / `lutState.test.ts` / `toneCurve.test.ts` / `histogram.test.ts` / `autoAdjust.test.ts`）し、WebGL/Canvas を使う実描画（LUT の GPU/CPU 出力ピクセル一致・強度 0 での無効化・プリセット適用・不正ファイル通知、シャープネスのエッジ増強と平坦部不変・ガンマの中間調上昇と白不変・モノクロの WYSIWYG・ビネットの両方向（減光 / 増光）と中心不変・グレインの決定性と GPU/CPU 粒一致（±2/チャンネル許容）と WebGL2 無効化時のフォールバック適用、トーンカーブの点追加によるプレビュー・出力の一致（WYSIWYG）とリセット復帰・輝度チャンネルの色味維持・WebGL2 無効化時の CPU 適用、ヒストグラムの輝度スパイク位置・露光量調整への追従・RGB/輝度切替・WebGL2 無効化時の表示、自動補正のオートレベルによるレンジ拡張とスライダー可視化・冪等性・自動ホワイトバランスのチャンネル平均等化・WebGL2 無効化時の適用、WB スポイトのクリック点基準の補正（gray-world なら逆符号になる 2 色画像での証明）・冪等性・分割位置不変・適用後の自動解除・Esc / 再クリック解除とヒント切替・WebGL2 無効化時の適用を含む）は E2E（`e2e/edit.spec.ts`。`.cube` フィクスチャは `e2e/helpers/fixtures.ts` で実行時生成）で検証する
- Canvas 依存の動作は Playwright E2E（`e2e/`）で実ブラウザ検証する。ダウンロード物はマジックナンバーや piexifjs のバイナリ解析で中身まで検証する（`e2e/metadata.spec.ts` 参照）。動的 import される重量ライブラリの実ブラウザロードも E2E で担保する（ファイル詳細モーダルの EXIF 表示 = exif-js の遅延ロードは `e2e/convert.spec.ts`）
- ツール連携（ハンドオフ）はツールメタの整合・送り先解決・File[] 変換・ストアセマンティクスの純粋ロジックを単体テスト（`handoff.test.ts`）し、実際のページ間の受け渡しは E2E（`e2e/handoff.spec.ts` 9 件: convert→crop / crop→convert / convert→metadata / metadata→convert（クリーニング結果の送出）/ edit→convert（編集 → 変換の中核フロー）/ convert→edit / convert→redact→metadata（レタッチ結果をメタデータ削除へ渡す投稿前の安全化フロー）の連鎖処理・AVIF 結果での導線非表示・戻る操作での二重取り込み防止）で検証する。ペイロードが in-memory のため E2E は `page.goto`（フルロード）ではなくアプリ内ボタンによるクライアントサイド遷移で検証する
- モザイク / ぼかしレタッチ（`/redact`）は領域リスト操作（`addRegion` / `updateRegionArea` / `removeRegion` の不変更新契約）・`resolveRegionsForIndex`・強度下限（`resolveMosaicBlockSize` / `resolveBlurRadius`）・`parseHexColor`・`clampRegionToImage`・RGBA バッファへのピクセル演算（塗りつぶしの指定色不透明化 / モザイクのブロック平均と端数ブロック / ぼかしの領域内クランプと決定性・領域外不変）を純粋ロジックとして単体テスト（`redactCore.test.ts`。あわせて `fileName.test.ts` に `appendFileNameSuffix`、`handoff.test.ts` に redact の受け取り・送り先解決を追加）し、実描画とダウンロード物のピクセル検証は E2E（`e2e/redact.spec.ts` 5 件: 領域なしで実行不可 + 対応形式に TIFF/HEIC を含まない・モザイク / ぼかしで赤青チェッカーが混合色に均される（不可読化。プレビュー / 出力とも）+ 領域外不変・塗りつぶしが指定色で完全に塗られる・複数領域の指定と個別削除。フィクスチャは `e2e/helpers/fixtures.ts` の `checkerPngFile`）で検証する
- アルファ非対応出力への背景合成（Issue #108）は、合成要否判定の純粋関数 `resolveFlattenBackground`（`conversionCore.ts`）を単体テスト（`imageConverter.test.ts`。JPEG は品質によらず背景色・PNG は低品質ティア（jpeg-roundtrip）のみ背景色・quality 未指定の PNG と WebP / AVIF は null = アルファ保持）し、実際の `fillRect` 合成〜エンコード結果は E2E で検証する: `e2e/convert.spec.ts` に透過 PNG → JPEG の白合成（Worker 経路 / メインスレッド強制経路 = addInitScript で `OffscreenCanvas.prototype.convertToBlob` を delete して Worker パイプラインを無効化）と透過 PNG → WebP のアルファ保持回帰の計 3 件、`e2e/edit.spec.ts` に透過 PNG の JPEG 出力での白合成 1 件。ダウンロード物のピクセル検証はブラウザ内デコードで RGB を読む共通ヘルパー `e2e/helpers/pixels.ts`（`readPixelFromBuffer`）を使う
- E2E のフィクスチャはバイナリを置かず `e2e/helpers/fixtures.ts` で実行時生成する（PNG は RGB / RGBA 両対応の `buildPng` を共通土台とし、透過検証用の `transparentPngFile`・redact 検証用の `checkerPngFile` を含む）
- E2E は本番同等の静的エクスポート（`npm run build` + `serve out`、ポート 3100）に対して実行される。ローカルで高速に回したい場合は `npm run dev -- --port 3100` を別途起動しておけば `reuseExistingServer` により再利用される（CI では常に build + 静的配信）
- PWA のプリキャッシュ判定・URL 変換・バージョン算出は Canvas 非依存の純粋関数 `precache.ts` に切り出して単体テスト（`precache.test.ts`）。Service Worker 本体の fetch/cache 制御は実ブラウザ動作なので Playwright E2E（`e2e/pwa.spec.ts`）で検証する：manifest link / theme-color / manifest 内容の確認と、SW 登録後に `context.setOffline(true)` で全ルート（/・/convert/・/crop/・/edit/・/redact/・/metadata/）がキャッシュから描画されること
- PWA の合否は Lighthouse の PWA カテゴリが Lighthouse 12 で廃止されたため、DevTools の Application パネルでの installability 確認と `pwa.spec.ts` のオフライン自動検証で担保する
- セキュリティヘッダー（CSP）は生成の純粋ロジック（`securityHeaders.ts` のインラインスクリプト抽出 `extractInlineScriptContents`（無加工の本文抽出 = CSP ハッシュの前提）・CSP 組み立て `buildPageCsp` / `buildCspRules`・マーカー区間の冪等マージ `mergeGeneratedRules`・Cloudflare Pages 制限検査 `validateHeadersContent`・ルール取り出し `parseCspForPath`）を単体テスト（`securityHeaders.test.ts`、21 件）し、CSP 強制下の実ブラウザ動作は E2E（`e2e/security-headers.spec.ts`）で検証する。E2E の webServer（serve out）は `out/_headers` を解釈しないため、postbuild が生成した実 CSP を `page.route` でドキュメントレスポンスへ注入し、全 6 ページ（/・/convert/・/crop/・/edit/・/redact/・/metadata/）の CSP ルール生成（script-src がハッシュベースで 'unsafe-inline' / 'unsafe-eval' なし）・表示 / ハイドレート / テーマ初期化（インラインスクリプトのハッシュ許可）・WebP 変換（WASM / Worker / blob プレビュー）が CSP 違反ゼロで動作すること、および `/*` のグローバルヘッダー定義を検証する（SW が HTML をキャッシュ配信すると route を通らないため `serviceWorkers: "block"` を使用。CSP は本番ビルドの postbuild でのみ生成されるため dev サーバー再利用時は skip = `pwa.spec.ts` と同方針）
