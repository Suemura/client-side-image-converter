# アーキテクチャ詳細

ディレクトリ / ファイル別の詳細な責務・関数一覧と、コア機能の詳細仕様。概要と横断規約は CLAUDE.md を参照。

## 主要ディレクトリ構造
- `src/app/` - Next.js App Router ページ
  - `convert/` - 画像フォーマット変換ページ。RAW ファイル投入時のみ「RAW 現像設定」パネル（`components/RawDevelopPanel.tsx` / `components/RawDevelopPanel.module.css`）を表示。露出補正（-2〜+3 EV）・ホワイトバランス（カメラ設定 / 自動 / 色温度指定 2000〜10000K）・ハイライト復元（クリップ / ブレンド / 再構築）を調整でき、パラメータは全 RAW ファイルへ一括適用。プレビューは先頭 RAW ファイルの half-size 再現像。デフォルト時は Issue #101 の固定現像と同一（Issue #132）
  - `crop/` - 画像トリミングツールページ（`components/CropSelector.tsx`（8方向ハンドル操作・比率固定リサイズ）と `components/CropToolbar.tsx`（アスペクト比・回転/反転・適用範囲）で構成）
  - `edit/` - 画像編集ページ（写真アプリ相当のライト/カラー/ディテール/効果の調整 + 自動補正（WB スポイト含む）+ トーンカーブ + LUT フィルタ + ヒストグラム表示、Issue #66 / #67 / #68 の第 1〜6 項目。`components/AdjustmentPanel.tsx`（先頭に自動補正グループ = オートレベル / 自動ホワイトバランスのワンショットチップボタン + WB スポイトのモードトグル（`aria-pressed`）、続いてライト/カラー/ディテール/効果のスライダー群（モノクロは 0/100 のチェックボックストグル、シャープネス / グレインは `ADJUSTMENT_UI_MIN` による 0 起点の片方向スライダー）。props `onAutoLevels` / `onAutoWhiteBalance` / `onToggleEyedropper` / `eyedropperActive` / `autoDisabled`）・`components/EditToolbar.tsx`（適用範囲・出力形式・EXIF 保持・全体リセット）・`components/CompareView.tsx`（分割 Before/After 比較・画像ナビ・WebGL レンダラ保持。編集後プレビューの描画完了ごとに縮小サンプリングした ImageData を渡す optional prop `onEditedFrame`、WB スポイト用の optional props `eyedropperActive` / `onEyedropperPick`（モード中は分割ドラッグの代わりにクリック点をソース自然座標で親へ渡す。crosshair カーソル + ヒント文切替）付き）・`components/ToneCurvePanel.tsx`（SVG トーンカーブエディタ。RGB マスター / 輝度のチャンネル切替チップ・クリックで点追加 → そのままドラッグ配置・ドラッグ移動・ダブルクリック削除（点の追加・ドラッグはプライマリポインタの左ボタンのみ）・編集前（カーブ適用前）の輝度ヒストグラム背景・チャンネル単位のカーブリセット。数式・点操作は `toneCurve.ts` へ委譲し座標変換とイベント処理に徹する）・`components/LutPicker.tsx`（LUT プリセットサムネイル一覧・「なし」・カスタムアップロード（.cube / HALD PNG）・適用強度スライダー・エラー通知）・`components/HistogramPanel.tsx`（調整・トーンカーブ・LUT 適用後のヒストグラムを SVG パスで表示。RGB / 輝度の切替チップ）で構成。ツール連携の送受信に対応（Phase 3 = Issue #72）: 受け取りは `useHandoffReceiver` + `HandoffNotice` の配線（受理形式は UPLOAD_FORMATS）、送出は編集結果の `ConversionResults` に `handoffOrigin="edit"` を指定した既存共用機構）
  - `redact/` - モザイク / ぼかしレタッチページ（Issue #98 Phase 1。プレビュー canvas 上で複数の矩形領域をドラッグ作成・ドラッグ移動・8方向ハンドルでリサイズ・× ボタンで個別削除し、モザイク / ぼかし / 塗りつぶしを焼き込む。`components/RedactSelector.tsx`（複数矩形セレクター。レタッチ焼き込み済みプレビュー canvas の表示 + 空き領域ドラッグで新規作成・矩形ドラッグで移動・8方向ハンドルでリサイズ・× ボタンで個別削除。座標変換は cropGeometry の `scaleCropArea` / `toDisplayArea` / `clampCropArea` を再利用）と `components/RedactToolbar.tsx`（隠し方のチップ切替 + モード連動パラメータ（モザイクの粗さ / ぼかしの強さの Slider・塗りつぶし色のカラーピッカー）+ 領域数表示 + 領域クリア）で構成。隠し方の設定はページ全体で 1 つ、領域は画像ごとに保持（←/→ ナビ連動。crop の「画像ごと」モード相当で「全画像一括」トグルはなし）。EXIF 保持オプション対応（レタッチ前画像が残る EXIF サムネイルは常時除去）。ツール連携の送受信に対応（受け取りは `useHandoffReceiver` + `HandoffNotice`（受理形式は UPLOAD_FORMATS）、送出は `ConversionResults` に `handoffOrigin="redact"`）。Phase 2（ローカル顔自動検出）は後続 Issue で対応予定）
  - `metadata/` - EXIF メタデータエディターページ（ツール連携の送受信に対応（Phase 2 = Issue #71）。受け取りは `useHandoffReceiver` + `HandoffNotice` の配線（受理形式は UPLOAD_FORMATS）、送出はタグ選択時に表示される「選択したメタデータを削除して次のツールへ」導線（i18n `handoff.cleanAndSendTo`。クリック時に `removeSelectedMetadata` を実行してから送出するダウンロード不要の経路。クリーニングは形式を変えないためファイル名は元名のまま））
  - `share/` - 共有シート（Web Share Target）の受け口ページ（Issue #105。インストール済み PWA でスマホの共有シートから共有された画像を受け取り、既存のハンドオフ送出 UI（`HandoffSend` の `origin="share"`）で行き先ツールを選んで引き継ぐ。mount 時に `readSharedPayload` で SW が保管したペイロードを読み取り（読み取りと同時に削除されるため、StrictMode の effect 二重実行は ref ガードで 1 回に限定）、受理外 MIME は `filterValidFiles` で除外して skipped 件数を表示する。空状態・非対応環境（`caches` なし）の表示あり。共有シート専用の機能的エンドポイントのため robots noindex + sitemap 除外（`next-sitemap.config.js` の `exclude`）。`layout.tsx`（metadata 定義用サーバーコンポーネント層）+ `page.tsx` + `share.module.css` で構成）
  - `upscale/` - AI 超解像（アップスケール）ページ。Real-ESRGAN の軽量モデル（realesr-general-x4v3、ONNX。公開 BSD-3-Clause）を ONNX Runtime Web（WebGPU 優先・WASM フォールバック）でブラウザ内実行し、JPEG / PNG / WebP / BMP を 2x / 4x に拡大する（Issue #100）。モデル・WASM ランタイムアセット（合計約 31MB）は初回実行時に取得して Cache Storage（`wic-model-cache`）へ保存。出力は元形式を維持し、`_upscaled` サフィックス付き。ツール連携（ハンドオフ）は convert 直後の送り先として組み込まれ、編集結果の最終アップスケールフローに対応。ページ固有の設定 UI は `UpscaleSettings.tsx`（2x/4x 倍率選択・EXIF 保持トグル）。バッチ処理は `upscaleRunner.ts` がワーカー `upscale.worker.ts` でタイル並列推論を制御し、キャンセル機能付き。長辺 4096px 超の画像は処理不可（`MAX_UPSCALE_INPUT_DIMENSION`）。`layout.tsx` + `page.tsx` + `upscale.module.css` + `components/UpscaleSettings.tsx` で構成
  - `manifest.ts` - PWA の Web App Manifest。Next が `/manifest.webmanifest` として静的出力し `<link rel="manifest">` を自動注入する（`dynamic = "force-static"`）。`share_target` エントリ（POST / multipart/form-data / files name="images"。accept は `resolveShareAcceptTypes()` による全受け取り可能ツールの MIME 和集合で HEIC / TIFF を含む）を含む（Issue #105。POST は SW が intercept する）
- `src/components/` - 再利用可能な React コンポーネント（PWA 関連: `ServiceWorkerRegister.tsx`（本番ビルドでのみ `/sw.js` を登録・UI なし）/ `InstallPrompt.tsx`（`beforeinstallprompt` を受けた控えめな A2HS 導線）。`Slider.tsx` はラベル・値表示・個別リセット付きの汎用レンジスライダー（編集ページの調整パネルで使用）。`MobileMenu.tsx`（モバイルレイアウト対応用のハンバーガーメニュー。`max-width: 768px` 以下で右スライドインドロワー（幅 300px）に収納される Navigation / GitHub リンク / ThemeSwitch / LanguageSwitch を管理。aria-current で現在ページを表示。Escape キーと背景幕クリックで閉じる）。`Results.tsx` は全ツール共用の結果表示コンポーネント（ファイル一覧・統計情報・ZIP ダウンロード・フォルダ直接保存（Chromium 系のみ）・ハンドオフ送出）で、Chromium 系ブラウザでは feature detection で「フォルダへ保存」ボタンを表示。ツール連携（ハンドオフ）関連: `HandoffSend.tsx`（結果を次のツールへ送る送出コントロール。`Results.tsx` の optional prop `handoffOrigin` 指定時に表示・送り先候補 0 件なら非描画・マウント時に送り先ルートを `router.prefetch`。`getFiles` は非同期 `() => File[] | Promise<File[]>` も受け付け（metadata のように送出クリック時に処理を実行するツール向け。空配列なら送出しない）、見出しの i18n キーを差し替える `labelKey`（既定 `handoff.sendTo`）と処理中無効化の `disabled`、非同期実行中の多重クリック防止（isSending）を備える。`origin` prop は `HandoffOrigin`（ツール id または共有シート受け口の "share"、Issue #105））/ `HandoffNotice.tsx`（受け取りページの到着バナー。`role="status"`・取り込めなかった件数の補足・閉じるボタン。共有シート起点（origin "share"）はツールメタを持たないため専用文言 `handoff.receivedFromShare` で表示））
- `src/contexts/` - React Context（`ThemeContext.tsx`（ライト/ダークテーマ）/ `HandoffContext.tsx`（`HandoffProvider` / `useHandoff`。ツール連携の共有ストアを配る。ストア本体は Next.js のクライアント遷移でルートレイアウトごと再マウントされても失われないようモジュールスコープのシングルトンで保持し、`usePathname` の変化を `store.onNavigate` へ通知する））
- `src/utils/` - コアユーティリティクラス
  - `imageConverter.ts` - 画像フォーマット変換処理（単体変換 `convertImage` と一括変換 `convertMultipleImages`。一括は対応環境で Worker プールへ委譲し、非対応環境ではメインスレッド逐次処理。`mode: "optimize"` のときは形式維持の最適化として `optimizeImage` に分岐し、`optimizeImage` を再エクスポートする）
  - `conversionCore.ts` - 変換のコア型（`ConversionFormat` / `ConversionMode`（"convert" / "optimize"） / `ConversionOptions`（`mode` / `rawDevelopParams`（Issue #132）を含む） / `ConversionResult` 等）と Canvas 非依存の純粋ロジック（`searchQualityForTargetSize` / `calculateTargetSize`、アルファ非対応出力へ合成する背景色を判定する `resolveFlattenBackground` と背景色定数 `FLATTEN_BACKGROUND_COLOR`（Issue #108。メイン / Worker / 編集の全エンコード経路が共有する唯一の真実））。メインスレッドと Worker で共有（`imageConverter.ts` から再エクスポートし既存 import 経路を維持）
  - `conversionResult.ts` - `ConversionResult` 組み立ての共通化（変換用 `buildConversionResult` / 最適化用 `buildOptimizeResult`（同一形式のため元ファイル名を維持）/ 編集用 `buildEditResult`（`_edited` サフィックス + 出力形式の拡張子。`originalFilename` は元名保持））。convertImage・ワーカープール・imageEditor で共用
  - `concurrency.ts` - 純粋な並行スケジューラ（`mapWithConcurrency` / `resolveConcurrency`）。同時実行数制限・入力順保持・continue-on-error・逐次進捗
  - `pngQuality.ts` - PNG 品質ティア判定（`pngQualityStrategy`。閾値 95/70）。メインスレッドと Worker で共有
  - `decodedImage.ts` - HEIC / TIFF / RAW デコード結果（RGBA 生ピクセル）の共有型 `DecodedImage`
  - `avifEncoder.ts` - AVIF エンコード処理（`@jsquash/avif` の WASM を動的 import。`HTMLCanvasElement | OffscreenCanvas` 両対応）
  - `optimizeCore.ts` - 画像最適化（形式維持の再圧縮）のコア型と Canvas / WASM / DOM 非依存の純粋ロジック（型 `OptimizeEngine` / `WebpEncoding`、形式→エンジンをディスパッチする `resolveOptimizeEngine` / `isOptimizableType`、no-worse-than-original 判定の `pickSmallerSize`、RIFF/VP8L/VP8/VP8X 走査で WebP のロスレス/ロッシーを判定する `detectWebpEncoding`）。メインスレッドと Worker で共有し単体テスト対象（`conversionCore.ts` と同方針）
  - `imageOptimizer.ts` - 形式維持の最適化のブラウザ側オーケストレーション（`@jsquash/oxipng` / `@jsquash/jpeg` / `@jsquash/webp` を動的 import。Canvas 非依存でメイン/Worker 共用の `optimizeImageBuffer`、メインスレッド用ラッパー `optimizeImage`）
  - `heicDecoder.ts` - HEIC/HEIF デコード処理（libheif WASM、動的 import。Canvas 非依存の `decodeHeicToImageData` を含む）
  - `tiffDecoder.ts` - TIFF デコード処理（`utif2` の純 JS デコーダー、動的 import。Canvas 非依存の `decodeTiffToImageData` を含む）
  - `rawDecoder.ts` - カメラ RAW（CR2 / CR3 / NEF / ARW / DNG / RAF / ORF 等）デコード処理（LibRaw の WASM ビルド `libraw-wasm`（LibRaw 0.22.1 + LCMS2 同梱、約 1.5MB）を動的 import。カメラ設定準拠のデフォルト現像（useCameraWb / 8bps）で開き、Canvas / DOM 非依存で Worker からも使える `decodeRawToImageData`（finally で `dispose()` し WASM メモリと内部 Worker を即時解放）と、メインスレッド用の `decodeRawToCanvas` を提供。heicDecoder / tiffDecoder と同型。Issue #132 で `decodeRawToImageData(buffer, params?, { halfSize? })` / `decodeRawToCanvas(file, params?)` に現像パラメータ引数を追加し、`buildLibRawSettings` の変換結果を `open()` の settings として渡すことで露出補正・ホワイトバランス・ハイライト復元を反映できるようにした）
  - `rawDevelopment.ts` - RAW 現像パラメータの純粋ロジック（Canvas / WASM 非依存・単体テスト対象）（Issue #132）。型 `RawDevelopParams`（`exposureEV` / `wbMode` / `kelvin` / `highlightMode`）、定数 `EXPOSURE_EV_MIN/MAX`（-2〜+3）/ `KELVIN_MIN/MAX`（2000〜10000K）/ `KELVIN_DEFAULT`（6500）/ `DEFAULT_RAW_DEVELOP_PARAMS`、デフォルト判定 `isDefaultRawDevelopParams`、色温度ケルビン → RGB 乗数への変換 `kelvinToWbMultipliers`（黒体放射 Tanner Helland 近似）、ユーザー向けパラメータを `LibRawSettings` へ変換する `buildLibRawSettings`（自動明るさ調整の有効/無効化 `noAutoBright`・露出補正の EV → expShift 変換・ホワイトバランスの mode 分岐・ハイライト復元モード指定）。パラメータは `ConversionOptions` へ持たせられ、`imageProcessing.worker.ts` で Worker 側の `decodeRawToImageData` へ渡される
  - `rawImage.ts` - RAW デコード結果の純粋変換ロジック（libraw-wasm の `imageData()` が返す生ピクセル（RGB / RGBA、8bit / 16bit）を RGBA 8bit の `DecodedImage` へ変換する `rawImageDataToRgba`（RGB はアルファ 255 補完・16bit は上位 8bit へ縮退（`>> 8`）・不正な colors / bits / データ長不整合は例外）と入力型 `RawImageDataLike`）。Canvas / WASM 非依存で単体テスト対象
  - `imageCropper.ts` - 画像トリミング処理（EXIF Orientation 補正 + 回転/反転をキャンバスに焼き込む `renderOrientedImage` / `createOrientedPreviewUrl`、`cropImage`（`transform` / `quality` 引数対応）、画像ごとの領域・変換を受け取る `cropImages`（`CropJob[]`）。`CropArea` / `CropTransform` は `cropGeometry.ts` から再エクスポート）
  - `cropGeometry.ts` - トリミングの座標計算に関する Canvas 非依存の純粋ロジック（型 `CropArea` / `CropTransform` / `CropState`、`ASPECT_RATIO_PRESETS`、`clampCropArea` / `scaleCropArea` / `toDisplayArea` / `fitAspectRatio` / `enforceAspectRatio` / `orientedSize` / `rotateRight` / `rotateLeft` / `resolveCropForIndex`）。CropSelector（表示座標）・crop ページ（自然座標保持）・imageCropper（出力）で共有し単体テスト対象
  - `redactCore.ts` - モザイク / ぼかしレタッチ（`/redact`）のコア型と Canvas 非依存の純粋ロジック（型 `RedactMode`（"mosaic" | "blur" | "fill"）/ `RedactStyle`（mode / mosaicBlockSize / blurRadius / fillColor。ページ全体で 1 つ保持し領域には持たせない）/ `RedactRegion`（id + `CropArea`）/ `RedactState`（`perImageRegions` のみ。領域指定は画像ごとが本質のため dual-store の共有側は持たない）、既定値 `DEFAULT_REDACT_STYLE`（弱いぼかしの復元攻撃リスク対策で既定はモザイク）、UI 範囲定数 `MOSAIC_BLOCK_SIZE_MIN/MAX`（8-64）/ `BLUR_RADIUS_MIN/MAX`（4-32）、領域リストの不変更新 `addRegion` / `updateRegionArea` / `removeRegion`（id 不在時は同一配列参照を返す契約）、`resolveRegionsForIndex`（`resolveCropForIndex` 踏襲）、領域サイズ相対の強度下限 `resolveMosaicBlockSize`（短辺 / 24）/ `resolveBlurRadius`（短辺 / 16。復元攻撃への構造的対策）、`parseHexColor`、境界クランプ `clampRegionToImage`、RGBA バッファへの in-place 適用 `applyRedactionsToImageData`（塗りつぶし = 指定色不透明 / モザイク = ブロック平均・端数ブロックは実画素数で平均・決定的 / ぼかし = 分離型ボックスブラー 3 パス・サンプリングは領域内クランプで領域外を読まない））。単体テスト対象

## コア機能
1. **画像フォーマット変換 / 最適化** - JPEG、PNG、WebP、AVIF、JPEG XL 形式への変換（品質制御付き。AVIF・JPEG XL は出力のみ対応）。目標ファイルサイズ (KB) を指定すると品質を二分探索して目標以下で最大品質の結果を出力する（JPEG / WebP のみ。達成不可時は最小サイズで出力し一覧に警告表示）。処理モードのラジオで「フォーマット変換」と「最適化（形式を維持）」を切り替えられ、最適化はフォーマットを変換せず入力と同じ形式のまま再圧縮してサイズを削減する（PNG は可逆・ピクセル不変、JPEG / WebP は高品質で再エンコード。削減できなければ元をそのまま採用。対応は PNG / JPEG / WebP のみで対応外は失敗通知。最適化モード時は変換用の設定（対象フォーマット・品質・目標サイズ・画像サイズ・EXIF 保持）を非表示にする）。HEIC/HEIF（iPhone 写真）と TIFF、カメラ RAW（.dng / .cr2 / .cr3 / .nef / .nrw / .arw / .raf / .orf / .rw2 / .pef / .srw）は変換ページのみ入力として受理（crop / edit / redact / metadata はブラウザがプレビュー描画できないため対象外）。RAW は LibRaw の WASM（`libraw-wasm`）でカメラ設定準拠のデフォルト現像（useCameraWb / 8bps。現像パラメータ UI なし・本格調整は変換後の /edit ハンドオフで行う想定）としてデコードし、EXIF 引き継ぎは HEIC / TIFF と同一挙動で対象外（フォローアップは Issue #128）。最適化（形式を維持）モードでは RAW は対応外（失敗一覧に表示）。変換・最適化に失敗したファイルは一覧で画面に通知される
2. **画像トリミング** - プレビュー付きのビジュアルトリミングインターフェース。アスペクト比プリセット（自由 / 1:1 / 16:9 / 4:3 / 3:2。自由以外は選択中のドラッグ・8方向リサイズ・移動が比率固定）、右/左 90° 回転・左右/上下反転、適用範囲の切り替え（「全画像一括」／「画像ごと」。画像ごとは ←/→ ナビゲーションと連動して画像単位に領域・回転/反転を保持・適用）に対応。プレビュー時に EXIF Orientation を `createImageBitmap(imageOrientation: "from-image")` で自動補正し回転/反転をキャンバスへ焼き込む（EXIF 保持時は出力の Orientation タグを 1 に正規化して二重回転を防ぐ）
3. **画像編集（ライト/カラー/ディテール/効果の調整 + 自動補正 + トーンカーブ + LUT フィルタ）** - 写真アプリ相当のライト/カラー/ディテール/効果の調整・自動補正（WB スポイト含む）・トーンカーブ・LUT（Look-Up Table）フィルタを非破壊で行い複数画像へまとめて適用できる `/edit` ページ（Phase 1 MVP = Issue #66、Phase 2 LUT = Issue #67、トーンカーブ / 自動補正 / WB スポイト / ディテール系 / モノクロ・ガンマ = Issue #68 の第 2〜6 項目）。ライト（露光量 / ガンマ / 輝度 / コントラスト / ハイライト / シャドウ / 白レベル / 黒レベル）とカラー（彩度 / 自然な彩度（Vibrance）/ 色温度 / 色合い（Tint）/ 色相 / モノクロ）、ディテール（シャープネス / 明瞭度）、効果（ビネット / グレイン）を各項目リセット + 全体リセット付きで調整する（モノクロは 0/100 のチェックボックストグルで、色温度 / 色合い / 色相がモノクロ化前に効くため B&W のカラーフィルタとして使える（彩度スライダーと独立）。シャープネス / グレインは 0 起点の片方向スライダー。グレインは座標ハッシュの決定的ノイズでプレビューと出力・GPU と CPU で同じ粒。ビネットは負値で周辺増光）。調整パネル先頭の「自動補正」（オートレベル / 自動ホワイトバランス）はワンショットボタンで、編集前ヒストグラムの統計から既存スライダー値を逆算してセットする（オートレベルは輝度ヒストグラムのパーセンタイルクリップ（片側 0.5%）で黒点・白点を推定して黒レベル / 白レベルを、自動ホワイトバランスは gray-world で色温度 / 色合いをセット。結果はスライダーに可視化されそのまま手動微調整・個別リセットでき、編集前統計基準のため同じ画像で再押下しても値は不変（冪等）。「すべてリセット」でリセットされる）。「WB スポイト」はトグルでモード化し、プレビュー上の「無彩色にしたい点」をクリックすると編集前ソースの 5×5 近傍平均から色温度 / 色合いを逆算してセットする（同じ点の再クリックで値不変（冪等）。クリック適用後は自動解除のワンショットで、Esc / トグル再クリック / 画像切替 / リストクリアでも解除。モード中は crosshair カーソル + ヒント文がスポイト用に切り替わり分割ドラッグは開始しない。Before/After のどちら側をクリックしても同じソース画素に写像される）。トーンカーブは RGB（マスター・全チャンネル同一適用）/ 輝度の 2 チャンネルを SVG エディタで編集でき（プロット領域のクリックで制御点を追加してそのままドラッグ配置・ドラッグで移動・ダブルクリックで削除。端点は x 固定で y のみ可動・最大 16 点。背景に編集前（カーブ適用前）の輝度ヒストグラムを重ねて表示）、チャンネル単位の「カーブをリセット」に対応する（全体リセットにも含まれる）。LUT フィルタは同梱プリセット（11 本）から選ぶか、`.cube`（1D / 3D）または HALD CLUT PNG をカスタムアップロードして適用でき、適用強度（0-100）で元色とのブレンド比を調整できる（適用順は ディテール → 調整 → トーンカーブ → LUT → ビネット / グレイン に固定）。プレビューは WebGL2 フラグメントシェーダでリアルタイム反映し、非対応環境は Canvas2D CPU パスへ自動フォールバック（プレビューと出力は同一の描画経路 `applyAdjustmentsToCanvas` を通すため結果が一致＝WYSIWYG）。分割スライダーの Before/After 比較、適用範囲トグル（全画像一括 / 画像ごと。crop と同じ dual-store で ←/→ と連動。調整・トーンカーブ・LUT フォールバック環境でも同一経路で表示）。出力は元形式維持（既定）または形式選択（JPEG/PNG/WebP/AVIF）で、EXIF 保持は AVIF 以外で有効（向き焼き込みに合わせ Orientation を 1 に正規化）。受理形式は crop と同じ UPLOAD_FORMATS（JPEG/PNG/WebP/BMP。HEIC/TIFF はプレビュー描画不可のため対象外）。当面はメインスレッド逐次処理（crop 同様）
4. **モザイク / ぼかしレタッチ** - 顔や写り込みなど見せたくない部分を隠す `/redact` ページ（Issue #98 Phase 1）。プレビュー上で複数の矩形領域をドラッグ作成・ドラッグ移動・8方向ハンドルでリサイズ・× ボタンで個別削除し、モザイク（ブロックサイズ指定）/ ぼかし（強度指定）/ 塗りつぶし（色指定）の 3 種で焼き込む。既定はモザイク（弱いぼかしには復元攻撃のリスクがあるため）で、大きな領域ではモザイク / ぼかしの強度が領域短辺相対の下限で自動的に引き上げられる（構造的な復元攻撃対策。UI に注記表示）。隠し方の設定はページ全体で 1 つ、領域は画像ごとに保持（←/→ ナビと連動。crop の「画像ごと」モード相当で「全画像一括」トグルはなし。領域未指定の画像はそのまま再エンコードして結果一覧に並ぶ）。EXIF Orientation は焼き込みで補正し、出力は元形式維持・`_redacted` サフィックス・EXIF 保持オプション対応（Orientation 正規化込み。EXIF 埋め込みサムネイル（IFD1）にはレタッチ前の画像が残るため EXIF 保持時も常時除去し、除去を保証できない場合は EXIF 引き継ぎを中止する = fail-closed。UI ヘルプに明記）。受理形式は crop と同じ UPLOAD_FORMATS（HEIC / TIFF はプレビュー描画不可のため対象外）で、領域が 1 件もない場合は実行不可。処理はメインスレッド逐次（crop 同様）。Phase 2（ローカル顔自動検出による領域の自動提案）は後続 Issue で対応予定
5. **EXIF メタデータ管理** - EXIF データの表示（JPEG / PNG / WebP に対応）、編集、選択的削除。GPS 位置は「削除」に加えて「市区町村レベルに丸める」（約 1km 精度で残す）を選択可能（GPS タグがある場合のみ UI 表示。丸めは JPEG のみ有効で、その他の形式は Canvas 再描画により全メタデータを削除）。変換・トリミング・編集・レタッチの「EXIF を保持」は JPEG / PNG / WebP 出力で有効（AVIF・JPEG XL は非対応）
6. **ツール連携（ハンドオフ）** - 変換・最適化 / トリミング / 画像編集 / レタッチ / メタデータ編集 / AI 超解像の結果をダウンロードせずに次のツールへ引き継いで続けて処理できる（Phase 1 = Issue #70 で convert ⇄ crop、Phase 2 = Issue #71 で metadata、Phase 3 = Issue #72 で edit、レタッチツール追加 = Issue #98 で redact、AI 超解像追加 = Issue #100 で upscale の送受信に対応し**全 6 ツールの相互連携**。「変換 → アップスケール → 編集」「編集 → 変換/最適化」「レタッチ → メタデータ削除 → 投稿」のフローが成立）。結果画面に「この結果を次のツールへ（ダウンロード不要で続けて処理）」の送り先ツール別ボタンを表示する（候補は「現在のツール以外」かつ「全結果 MIME を受理できるツール」のみ。例: edit の AVIF 出力を含め AVIF 結果は受理できるツールがないため導線ごと非表示）。metadata の送出はタグ選択時に表示される「選択したメタデータを削除して次のツールへ」導線で、クリック時にクリーニング（`removeSelectedMetadata`）を実行してから送出する（ダウンロードしない経路。ファイル名は元名のまま）。受け取りページには到着バナー（`role="status"`。「変換の結果 N 件を引き継ぎました」+ 取り込めなかった件数の補足 + 閉じるボタン）を表示し、引き継ぎは新規アップロードと同じフィルタ（MIME・重複除外・`MAX_INPUT_FILES`（200 件）上限）を通る。ペイロードは File 実体のみを in-memory 保持（ObjectURL・ストレージ・URL 非経由。リロードで消える）し、送出時に元ページの結果はクリアされ ObjectURL は revoke される。ファイル名は既存命名のまま連鎖する（例: sample.png → sample.jpeg → sample_cropped.jpeg）。共有シート受け口 `/share`（Issue #105）も送出起点（origin "share"）として同じ機構で任意のツールへ引き継ぐ
7. **バッチ処理と結果の保存** - 複数画像の一括処理（ドラッグ&ドロップ / ファイル選択 / クリップボード貼り付け / フォルダドロップで投入）。一度に取り込める件数はハード上限 `MAX_INPUT_FILES`（200 件）までで、フォルダの再帰取込や複数選択で上限を超えた分は取り込まず非ブロッキングの警告を表示する（大量投入時のフリーズ/メモリ圧迫を防ぐため）。変換ページは対応環境で Web Worker + OffscreenCanvas のプール（`navigator.hardwareConcurrency` を上限に並列）で処理し UI をブロックしない（非対応環境はメインスレッド逐次処理にフォールバック）。結果の保存は ZIP ダウンロード（全ツール対応）または Chromium 系ブラウザでのフォルダ直接保存（File System Access API の `showDirectoryPicker` で選択したローカルフォルダへ圧縮なしで書き込み。同名衝突は上書きせず ZIP / ハンドオフと同じ連番規則で自動一意化。永続化しない（プライバシー原則）。Issue #106）で選択できる
8. **プライバシーファースト** - Canvas API / WASM / WebGL を使用したクライアントサイドでの全処理。本番配信では全パス共通のセキュリティヘッダー（X-Frame-Options / nosniff / Referrer-Policy / Permissions-Policy / HSTS）とページ別のハッシュベース Content-Security-Policy を付与する（Issue #111。生成の仕組みは `docs/PATTERNS.md` を参照）
9. **PWA（オフライン対応・インストール可能）** - Web App Manifest とビルド時生成の Service Worker により、一度アクセスすれば JS/CSS/フォント/WASM/HTML/アイコンがプリキャッシュされ、オフラインでも全機能が動作する。対応ブラウザ（主に Chromium 系）ではホーム画面/デスクトップへインストール可能（`beforeinstallprompt` を受けた控えめな導線を表示）。インストール済み PWA（Chromium 系 + OS の共有シート統合）ではスマホの共有シートから画像を直接受け取れる（Web Share Target、Issue #105。manifest の share_target への POST を SW が intercept して一時キャッシュへ保管し、受け口ページ `/share` が読み取りと同時に削除して既存ハンドオフで任意のツールへ引き継ぐ。受理 MIME は全受け取り可能ツールの和集合で HEIC / TIFF 含む。非対応環境の既存動作に影響なし）
