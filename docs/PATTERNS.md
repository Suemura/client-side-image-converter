# 重要な実装パターン

コードから読み取りにくい設計判断・不変条件の詳細。該当領域に触れる前に読むこと。概要は CLAUDE.md の「重要な設計原則」を参照。

- 画像処理はクライアントサイド操作のために Canvas API を使用
- 変換ページのバッチ処理は Web Worker + OffscreenCanvas のワーカープール（`src/workers/`）で並列実行する。`navigator.hardwareConcurrency` を上限に Worker を起動し、デコード〜エンコード〜EXIF 挿入まで Worker 内で完結させメインスレッド（UI）をブロックしない。AVIF の WASM エンコード（従来メインスレッドで同期実行し UI をフリーズさせていた）も Worker 内で実行する。OffscreenCanvas / Worker / createImageBitmap 非対応環境（`isOffscreenPipelineSupported()` で判定）ではメインスレッド逐次処理にフォールバックし、Worker が個別に失敗・クラッシュした場合も当該ファイルのみ `convertImage` で再試行する。メインスレッドと Worker が共有する型・純粋ロジックは Canvas 非依存の `conversionCore.ts` / `concurrency.ts` / `pngQuality.ts` に集約し、Worker バンドルへの Canvas コード混入を防ぐ
- アルファ非対応出力への背景合成（Issue #108）: Canvas はアルファ 0 のピクセルが RGBA(0,0,0,0) のため、透過画像を JPEG 等へそのままエンコードすると透過部分が黒くなる。`drawImage` の前に白背景（`FLATTEN_BACKGROUND_COLOR` = "#ffffff"）を `fillRect` で合成して防ぐ。合成要否の判定は `conversionCore.ts` の純粋関数 `resolveFlattenBackground(format, quality?)` を「唯一の真実」とし、メインスレッド変換（`imageConverter.ts`）・Worker 変換（`imageProcessing.worker.ts`）・編集の出力（`imageEditor.ts`）の全エンコード経路がこれをミラーする（WYSIWYG）。JPEG は常に背景色、PNG は quality 指定時かつ `pngQualityStrategy(quality) === "jpeg-roundtrip"`（中間 JPEG 経由で透過が失われるティア）のみ背景色、それ以外（PNG lossless / compressed・WebP・AVIF）は null（アルファ保持）。PNG をネイティブエンコードする経路（編集ページの PNG 出力等）では quality を渡さないこと（渡すと低品質ティアで不要な合成が起きる）
- AVIF・JPEG XL エンコードのみ Canvas の `toBlob` が全ブラウザ未実装のため、`@jsquash/avif` / `@jsquash/jxl`（WASM）を動的 import で使用（変換実行時のみロードされ、初期バンドルに影響しない。処理はブラウザ内で完結）
- 最適化（形式を維持）モードはフォーマットを変換せず、入力と同じ形式のまま再圧縮してサイズを削減する（Issue #61）。エンジンは形式ごとに動的 import で使い分ける: PNG は `@jsquash/oxipng`（バイト列→バイト列の真の可逆最適化・ピクセル不変・Canvas 不要）、JPEG は `@jsquash/jpeg`（mozjpeg。progressive + trellis 量子化での高品質再エンコードで実質劣化最小）、WebP は `@jsquash/webp`（ロスレス VP8L 入力はロスレス再エンコード、ロッシー入力は高品質ロッシー再エンコード）。いずれも最適化後が元以上なら元をそのまま採用する（no-worse-than-original）。メタデータの扱いは codec の能力に応じて非対称: JPEG は decode 時に `preserveOrientation: true` で EXIF Orientation をピクセルへ焼き込み（crop 経路と同じ WYSIWYG。タグ無し再エンコードでも正しい向きで表示される）、その他の EXIF は再エンコードにより除去される。WebP は `@jsquash/webp` が decode 時に Orientation を補正できず再エンコードで EXIF も失われるため、EXIF を持つ WebP は最適化せず元を採用して向きの崩れ・メタデータ喪失を防ぐ（EXIF 無しの WebP のみ再圧縮）。PNG は oxipng のバイト列最適化のためメタデータを保持する。対応形式は PNG / JPEG / WebP のみで、対応外（BMP / TIFF / HEIC / AVIF 等）は失敗として一覧通知される。jsquash はバッファを直接処理し OffscreenCanvas が不要なため、バッチは `Worker` があれば（OffscreenCanvas 非対応環境でも）ワーカープールで各形式のまま並列最適化し（混在バッチ可）、Worker 不在・個別失敗時はメインスレッドの `optimizeImage` にフォールバックする。形式→エンジンのディスパッチ・no-worse-than-original 判定・WebP のロスレス/ロッシー判定は Canvas 非依存の純粋関数 `optimizeCore.ts`（単体テスト対象）に集約し、jsquash を動的 import するブラウザ側オーケストレーションは `imageOptimizer.ts`（メイン/Worker 共用の `optimizeImageBuffer`・メインスレッド用 `optimizeImage`）が担う。UI の処理モード切替は `src/app/convert/components/ConversionSettings.tsx`（文言は i18n `convert.mode` / `convert.modeConvert` / `convert.modeOptimize` / `convert.optimizeDescription` など）
- HEIC/HEIF のデコードは libheif の WASM ビルド（`heic-decode` + `libheif-js`）を使用し、動的 import により HEIC 変換時のみロードする（初期バンドルに影響なし）
- TIFF のデコードは `utif2`（純 JS の TIFF デコーダー）を使用し、動的 import により TIFF 変換時のみロードする（初期バンドルに影響なし。マルチページ TIFF は先頭ページのみ対応）
- カメラ RAW（.dng / .cr2 / .cr3 / .nef / .nrw / .arw / .raf / .orf / .rw2 / .pef / .srw）のデコードは LibRaw の WASM ビルド `libraw-wasm`（LibRaw 0.22.1 + LCMS2 同梱、約 1.5MB）を使用し、動的 import により RAW 変換時のみロードする（Issue #101。初期バンドルに影響なしだが PWA のプリキャッシュはチャンク分増える）。変換ページ（/convert）のみ入力として受理し、現像はカメラ設定準拠のデフォルト（useCameraWb / 8bps 出力。現像パラメータ UI は持たず、本格的な調整は変換後の /edit ハンドオフで行う想定）。判定は他形式と異なり**拡張子優先**（`isRawFile`）——RAW は TIFF ベースの形式が多く、OS / ブラウザによって MIME が image/tiff・application/octet-stream・空文字で誤報告されるため、HEIC / TIFF の matchesFormat 方式（MIME 優先 + 拡張子フォールバック）では image/tiff 誤報告の NEF / DNG を utif2 に流してしまう。同じ理由で **RAW 分岐は必ず isTiffFile より前に置く**不変条件がある（`imageConverter.ts` の `convertImage` と `imageProcessingPool.ts` の `detectDecodeKind` の両方）。バッチ並列の Worker 経路（`DecodeKind` の "raw"）でも `decodeRawToImageData` を使うが、libraw-wasm は内部で自前の Worker を生成するため**ネスト Worker**になる（非対応環境では例外になり、既存フォールバックで当該ファイルのみメインスレッド `convertImage` で再試行される）。`generateThumbnail`（`imageUtils.ts`）は RAW なら即 null を返す（ブラウザの Image はデコードできず onerror で null になるだけなので、数十 MB のファイルを readAsDataURL で無駄にメモリ展開しない）。対応形式ラベルは RAW の MIME 群（10 種）を単一トークン "RAW" へ集約し（`formatAcceptedTypesLabel`）、ファイルタイプバッジは拡張子で表示する（`getFileTypeBadgeLabel`。"X-ADOBE-DNG" のような冗長ラベル回避）。EXIF 引き継ぎは HEIC / TIFF と同一挙動で対象外（フォローアップは Issue #128）
- HEIC / TIFF は MIME タイプが特定されない環境があるため、拡張子（.heic/.heif/.tif/.tiff）によるフォールバック判定を行う（`fileUtils.ts` の `FORMAT_EXTENSION_FALLBACKS`。`isHeicFile` / `isTiffFile` の判定と input の accept 属性の両方で使用。RAW も `FORMAT_EXTENSION_FALLBACKS` に拡張子グループを持つ（accept 属性・受理判定用）が、`isRawFile` の判定は上記のとおり拡張子優先）
- 画像の投入は全ページ共通の `FileUploadArea` に集約し、ドラッグ&ドロップ / ファイル選択 / クリップボード貼り付け（Ctrl/Cmd+V、`usePasteImages` フック）/ フォルダドロップ（`directoryReader.ts` による再帰走査）のいずれも共通関数 `addFiles`（`filterValidFiles` による MIME フィルタ → `addUniqueFilesWithLimit` による重複除外 + 上限件数 `MAX_INPUT_FILES`（200 件）での切り詰め）に合流させる。上限超過時は超過分を取り込まず、`role="alert"` の非ブロッキング警告ボックス（i18n `fileUpload.limitExceeded`）を表示する（件数の減少＝削除・クリアでリセット。判定は `fileUtils.ts` の `shouldClearLimitWarningOnDecrease`。「件数 < 上限」だけで消すと、走査打ち切り警告を追加操作直後の再描画で消してしまうため件数の減少を条件にする）。フォルダドロップは drop イベント中に `webkitGetAsEntry()` を同期取得してから再帰走査する（非対応環境は `dataTransfer.files` にフォールバック）。走査は `collectFilesFromEntries(entries, { maxFiles: MAX_INPUT_FILES + 1, accept })` で有界化し（`accept` は各ページの受理形式判定 `isAcceptedFileType`）、`readEntries` を 1 バッチずつストリーミング処理することでフラットな巨大フォルダでも Entry ハンドルが無制限に積み上がらないようにする。走査自体が上限で打ち切られた場合は戻り値の `reachedLimit` を `addFiles(files, collectionTruncated)` に渡し、重複除外後の切り詰め（`truncated`）と OR で取りこぼし警告に連動させる。クリップボード取り込みは `fileUtils.ts` の `getFilesFromClipboardData`（`.files` 優先、無ければ `.items` の `kind === "file"`）
- ツール連携（ハンドオフ）は純粋ロジック `handoff.ts`（単体テスト対象）と React への配線（`HandoffContext.tsx` / `useHandoffReceiver.ts` / `HandoffSend.tsx` / `HandoffNotice.tsx`）に分離する（Phase 1 = Issue #70、Phase 2 = Issue #71、Phase 3 = Issue #72 で convert / crop / edit / metadata、レタッチツール追加 = Issue #98 で redact が加わり全 5 ツールの送受信に対応）。ペイロード（File[]）は「1 回のナビゲーションだけ生存」するストア（`createHandoffStore`）で受け渡し、ストア本体はモジュールスコープのシングルトンに置く（Next.js のクライアント遷移ではタイミングによりルートレイアウトごと再マウントされることがあり、Provider の ref 保持では送出直後のペイロードが失われるため）。即時クリアの consume-once にはせず到着ページの間は冪等に consume できるようにし（App Router の二重マウントによる取りこぼし防止）、一方で送出元ページには配送せず、到着後に別パス名から consume / onNavigate された時点で破棄して戻る操作での二重取り込みを防ぐ。`HandoffSend` はマウント時に送り先ルートを `router.prefetch` する（未プリフェッチの `router.push` は RSC 取得失敗時にフル MPA ナビゲーションへフォールバックし、in-memory のペイロードがドキュメントリロードで失われるため）。受け取り側（`useHandoffReceiver`）は新規アップロードと同じフィルタ（`filterValidFiles` → `addUniqueFilesWithLimit`）を通してからページの投入関数へ渡す。ツールのメタ定義（パス・ラベルキー・受理形式・受け取り可否）は `HANDOFF_TOOLS` を Navigation と共有する単一の真実とする。metadata のように送出クリック時に処理を実行するツールは `HandoffSend` の非同期 `getFiles`（`() => File[] | Promise<File[]>`）で対応する（`labelKey` で見出しを差し替え・`disabled` と isSending で処理中の多重実行を防止・空配列なら送出しない）。UI 文言は i18n `handoff.*`
- トリミングの座標計算（境界クランプ・表示⇔自然座標変換・アスペクト比の当てはめ/強制・回転後寸法・画像ごとの領域/変換の解決）は Canvas / DOM 非依存の純粋関数 `cropGeometry.ts` に集約し、CropSelector（表示座標での操作。`aspectRatio` prop で比率固定リサイズ）・crop ページ（自然座標での state 保持と適用範囲トグル）・`imageCropper.ts`（出力）で共有して単体テストする。回転/反転と EXIF Orientation は `renderOrientedImage` でプレビューと出力の双方に同一ロジックで焼き込み（WYSIWYG）、EXIF 保持時のみ `normalizeExifForBakedImage` で出力 TIFF の Orientation を 1 に揃え、実ピクセル寸法タグ（PixelXDimension/PixelYDimension）が存在すれば出力寸法へ更新する。トリミングツールバー UI は `src/app/crop/components/CropToolbar.tsx`（文言は i18n `crop.*`）
- モザイク / ぼかしレタッチ（`/redact`、Issue #98 Phase 1）はピクセル演算（モザイク = ブロック平均・ぼかし = 分離型ボックスブラー 3 パス・塗りつぶし = 指定色不透明）と領域リスト管理を Canvas 非依存の純粋ロジック `redactCore.ts`（単体テスト対象）に集約し、Canvas との橋渡しは `imageRedactor.ts` が担う（cropGeometry / imageCropper と同じ分離方針）。プレビュー（`RedactSelector`）と出力（`redactImage`）はともに `renderRedacted`（領域ごとに getImageData → 純粋関数適用 → putImageData。全面ではなく領域単位で読むため処理量は領域面積に比例）を通して WYSIWYG を担保する。既定モードはモザイク（弱いぼかしには復元攻撃のリスクがあるため）とし、モザイク / ぼかしの強度は領域短辺相対の下限（`resolveMosaicBlockSize` = 短辺 / 24・`resolveBlurRadius` = 短辺 / 16）で自動的に引き上げる構造的対策を持つ（モザイクはブロック平均の決定的演算（乱数不使用）・ぼかしのサンプリングは領域内クランプで領域外の画素を読まず書き換えない）。領域指定は画像ごとが本質のため `RedactState` は per-image ストアのみ（dual-store の共有側はなし。解決は `resolveScopedValueForIndex` を踏襲した `resolveRegionsForIndex`）、隠し方の設定（`RedactStyle`）はページ全体で 1 つ（領域別スタイルが必要になったら `RedactRegion` へ optional フィールドを足して拡張する）。矩形操作の座標変換は `cropGeometry.ts`（`scaleCropArea` / `toDisplayArea` / `clampCropArea`）を再利用する。出力経路は `cropImage` と同型（`renderOrientedImage` → 焼き込み → 元形式 toBlob → `normalizeExifForBakedImage` + `insertExifIntoBlob`）で `CropResult` 互換を返すため、結果一覧（`ConversionResults`）とハンドオフ（`cropResultsToFiles`）をそのまま再利用できる。ただし EXIF 保持時は `stripThumbnail: true` を渡し、レタッチ前の縮小画像が残る EXIF 埋め込みサムネイル（IFD1）を常時除去する（除去を保証できない場合は例外になり EXIF 引き継ぎ自体を中止する fail-closed。「隠したはずの内容がサムネイル経由でリークしない」ことをツールの中核保証とする）。出力ファイル名は `fileName.ts` の `appendFileNameSuffix` による `_redacted` サフィックス。UI 文言は i18n `redact.*`
- 画像編集（ライト/カラー/ディテール/効果の調整 + トーンカーブ + LUT フィルタ）はプレビュー・出力とも同一の `applyAdjustmentsToCanvas`（`webglImageRenderer.ts`）を通す（WYSIWYG）。既定は WebGL2 フラグメントシェーダ（`adjustmentShader.ts`）でリアルタイム描画し、`createAdjustmentRenderer` が null を返す環境（WebGL2 非対応）は Canvas2D + `effects.ts`（`detailDeltaAt` / `vignetteFactorAt` / `grainNoiseAt`）+ `applyAdjustmentToPixel`（`adjustments.ts`）+ `applyToneCurveToPixel`（`toneCurve.ts`）+ `applyLutToPixel`（`lutParser.ts`）の CPU パスへフォールバックする。GPU（GLSL）と CPU（TS）が結果一致する構造的担保として、ピクセル単位の色調整数式は `adjustments.ts` の `applyAdjustmentToPixel` を唯一の真実とし、GLSL 側は同じ順序・同じ係数・同じクランプ位置でミラーする（キー→uniform 名の対応表 `ADJUSTMENT_UNIFORMS` も単一の真実として配線・アップロード・単体テストのガードが共有）。バッチ（`editImages`）は WebGL レンダラを 1 個だけ生成して全ファイルで使い回し、finally で必ず dispose する（コンテキスト枯渇を防ぐ。crop 同様メインスレッド逐次処理で、Worker + OffscreenCanvas 化は将来の最適化）。調整は非破壊で `AdjustmentState`（UI 単位 [-100,100]）として保持し、書き出し時に `normalizeAdjustments` で [-1,1] に正規化してから適用する。適用範囲（全画像一括 / 画像ごと）は汎用フック `useApplyScopeStore` で管理する。UI は `AdjustmentPanel`（`ConversionSettings` の不変更新契約を踏襲）/ `EditToolbar` / `CompareView` / `ToneCurvePanel` / `LutPicker` / `HistogramPanel`（文言は i18n `edit.*` / `edit.toneCurve.*` / `edit.lut.*` / `edit.histogram.*`。適用範囲・画像ナビは `crop.*` を共用、プレビュー生成失敗は `ErrorNotice` コンポーネント（`src/components/ErrorNotice.tsx`）で通知）。ヒストグラム（RGB / 輝度）は調整・トーンカーブ・LUT 適用後のプレビューを反映する: `CompareView` が編集後キャンバス（GPU / CPU フォールバックのどちらでも転写済みの 2D キャンバス）を `resolveHistogramSampleSize` で縮小 `drawImage` → `getImageData` した ImageData を optional prop `onEditedFrame` で渡し（連続再描画は rAF で 1 フレーム 1 回に間引き・サンプリング用キャンバスは ref で使い回し・コールバックは ref 経由で参照し GPU 再描画を誘発しない）、page 側が `histogram.ts` の `computeHistogram` で 256 ビンの R/G/B/輝度に集計して `HistogramPanel`（SVG パス描画・RGB/輝度チップ）が表示する（「リストをクリア」でリセット）
- LUT フィルタ（色変換フィルタ）は写真編集の標準順に従い調整（`applyAdjustmentToPixel`）の**後段**に適用する（Issue #67, Phase 2）。GPU（`adjustmentShader.ts` の GLSL）と CPU（`lutParser.ts` の `applyLutToPixel`）はドメイン正規化 → 3D テクスチャのトライリニア lookup → 強度ブレンドを同順序でミラーし WYSIWYG を担保する（`applyAdjustmentToPixel` を唯一の真実とするのと同方針。LUT サンプラー名 `LUT_SAMPLER` と uniform 名 `LUT_UNIFORMS` を配線の単一の真実にして GLSL 生成・アップロード・単体テストのガードが共有）。`.cube`（1D / 3D）と HALD CLUT はパース時に常に正規化済み 3D `LutData` へ揃える（1D は分離可能変換として 3D へ厳密展開）ため下流は単一の 3D グリッドだけを扱う。GPU は LUT を WebGL2 の 3D テクスチャ（内部形式 RGB8・LINEAR サンプリングがトライリニア補間を担う）としてアップロードし、既定は恒等 LUT を入れて LUT 未選択でも常時 complete に保つ。パース・CPU 適用の純粋ロジックは `lutParser.ts`（単体テスト対象）、選択状態は `lutState.ts`（単体テスト対象）、fetch / FileReader / Canvas デコードのブラウザ依存の橋渡しは `lutLoader.ts`（E2E 検証）に分離する。`AdjustmentState` は不変のまま、LUT は並列の軽量選択 state（`lutId` + `strength`）として持ち、重いデータ本体は edit ページの ref レジストリ（`lutId → LutData`）で参照する。プリセットの実データは `public/luts/*.cube` を動的 fetch し（初期バンドル非影響・モジュール内キャッシュ）、依存追加はなし（WebGL2 / Canvas はブラウザ標準）
- トーンカーブ（RGB マスター / 輝度チャンネル）は調整（`applyAdjustmentToPixel`）の後・LUT の前に適用する（Issue #68 の第 2 項目。適用順は ディテール → 調整 → トーンカーブ → LUT → ビネット / グレイン に固定）。制御点列（正規化 [0,1]²・端点 x 固定）を Fritsch–Carlson の単調 3 次 Hermite 補間で 256 エントリの 1D LUT へ焼成し（`buildToneCurveTable`。RGBA インターリーブの Float32Array(256×4) で `.rgb`=各チャンネル（現状マスター同値・将来 R/G/B 個別カーブへ拡張可能）/ `.a`=輝度）、GPU は 256×1 RGBA8 テクスチャ（TEXTURE2・既定は恒等）の LINEAR サンプリング + テクセル中心補正、CPU は同式の floor+lerp（`sampleCurveTable`）で lookup して WYSIWYG を担保する（テーブルは焼成時に 8bit 量子化済みのため GPU テクセルと CPU 参照値も同一）（サンプラー名 `CURVE_SAMPLER` と uniform 名 `CURVE_UNIFORMS` を配線の単一の真実にするのは LUT と同方針）。輝度カーブは既存トーン調整と同じ加算シフト方式（`curve(luma) - luma` を全チャンネルへ加算。luma は Rec.709）。恒等カーブはテーブルを焼成せず null を渡してサンプリング自体をスキップする（出力時は同一 `ToneCurveState` のテーブルを Map で重複焼成しない）。点操作・補間・焼成・CPU 適用は純粋ロジック `toneCurve.ts`（単体テスト対象）に集約し、`ToneCurvePanel` は座標変換とイベント処理に徹する。適用範囲（全画像一括 / 画像ごと）は調整・LUT と同じトグルで汎用フック `useApplyScopeStore` により管理される
- 自動補正（オートレベル / 自動ホワイトバランス / WB スポイト）は新しい描画パイプライン段を追加せず、編集前の統計から既存の調整スライダー値を逆算してセットする「統計 → スライダー値」方式（Issue #68 の第 3・4 項目。描画パイプライン（`applyAdjustmentToPixel` / GLSL / `EditJob` / `webglImageRenderer.ts` / `imageEditor.ts`）は一切不変）。オートレベルは編集前ヒストグラム（トーンカーブ背景用に分離済みの `sourceHistogram`）の輝度パーセンタイルクリップ（片側 0.5% = `AUTO_LEVELS_CLIP_RATIO`）で黒点・白点を推定し、パイプラインのトーンマスク重み（`blacksToneWeight` / `whitesToneWeight`）で逆算した blacks / whites をセットする。WB の逆算式は「点の色 → temperature / tint」の `computeWhiteBalanceForNeutralPoint` を単一の真実とし、自動ホワイトバランス（gray-world）はチャンネル平均をこの関数へ渡す特殊形、WB スポイトはクリック点の近傍平均を渡す形で共有する（`TEMPERATURE_SHIFT` / `TINT_SHIFT` 等の逆算係数は `adjustments.ts` が輸出し `applyAdjustmentToPixel` との整合を単一の真実として保つ）。WB スポイトは `AdjustmentPanel` のトグル（`aria-pressed`）でモード化し、プレビューのクリック点を `displayPointToSourcePixel` でソース自然座標へ写像 → **編集前ソース**（`previewSource`。getImageData で読むため型を `HTMLCanvasElement` に絞って保持）の 5×5 近傍（`WB_SAMPLE_RADIUS` / `clampSampleWindow` で読み出しを有界化）を `averageRgb` で平均して逆算する（編集後キャンバスから読むと補正結果が次のサンプル値へ混入するフィードバックになるため必ず編集前ソース基準。同じ点の再クリックで値不変 = 冪等）。クリック適用後は自動解除（ワンショット）で、Esc / トグル再クリック / 画像切替 / リストクリアでも解除。モード中は crosshair カーソル + `CompareView` のヒント文がスポイト用に切り替わり分割ドラッグを開始しない（Before/After の両キャンバスは同寸重畳のためどちら側のクリックも同じソース画素に写像）。結果はスライダーに可視化され手動微調整・個別リセットの出発点になり（Lightroom の Auto と同方針）、編集前統計基準のため同じ画像で再実行しても値は不変（冪等）。書き込みはいずれも `setCurrentAdjustments` 経由のため適用範囲（全画像一括 / 画像ごと）の dual-store 挙動は手動スライダー操作と完全に同一で、「すべてリセット」でリセットされる。純粋ロジックは `autoAdjust.ts`（単体テスト対象）、UI は `AdjustmentPanel` 先頭の「自動補正」グループ（i18n `edit.auto.*`。編集前統計が未算出の間はボタンを無効化）
- ディテール（シャープネス / 明瞭度）・ビネット / グレイン・モノクロ / ガンマ（Issue #68 の第 5・6 項目）は既存の単一シェーダパイプラインの拡張として実装する。パイプラインは「ディテール（ソース直後）→ 調整（露光 → ガンマ → 輝度 → コントラスト → トーン → 温度/色合い → 彩度 → vibrance → 色相 → モノクロ）→ トーンカーブ → LUT → ビネット → グレイン → 最終クランプ」に固定（ディテールは `texelFetch` によるソース近傍タップで近傍参照がソーステクスチャだけで完結するため、マルチパス / 中間 FBO 不要の単一シェーダのまま）。ガンマ / モノクロは per-pixel 色変換のため `applyAdjustmentToPixel`（手順 1b / 9b）へ、近傍参照（ディテール = 輝度ベースの unsharp mask。シャープネスはストライド 1 の小半径、明瞭度は解像度適応の整数ストライド `clarityStride` + 中間調マスク）と画素位置依存（ビネット / グレイン）の効果は `effects.ts` へ置き、GLSL は同じ順序・同じ係数（`SHARPNESS_GAIN` 等の係数定数と `GAUSS3_TAPS` カーネルをビルダーが埋め込み。明瞭度ストライドは `EFFECT_UNIFORMS` の `u_clarityStride` で配線）でミラーする。グレインは `Math.random` を使わず lowbias32 整数ハッシュ（TS は `Math.imul`、GLSL は uint 演算でビット同一・上位 24bit のみ float 化）による決定的ノイズで、プレビューと出力・GPU と CPU で同じ粒になる（E2E は ±2/チャンネルで粒一致を検証。fp32/fp64 の丸め差が理論上残るため LUT の GPU/CPU 厳密一致とは基準を分ける）。`AdjustmentState` への 6 キー追加（gamma / monochrome / sharpness / clarity / vignette / grain）だけで dual-store（一括 / 画像ごと）・全体リセット・`ADJUSTMENT_UNIFORMS` の自動配線・`clampAdjustments` の欠損キー 0 補完（後方互換）が既存機構のまま機能するため、page.tsx / `EditJob` / `imageEditor.ts` / `CompareView` は変更なし。値 0 の項目は GPU（uniform 分岐）/ CPU（フラグガード）とも計算を完全スキップし既存プレビュー性能は不変。UI は `AdjustmentPanel` の「ディテール」「効果」グループ（i18n `edit.detail` / `edit.effects` / `edit.sharpness` / `edit.clarity` / `edit.vignette` / `edit.grain` / `edit.gamma` / `edit.monochrome`）
- EXIF データ処理は保存に `piexifjs`、読み取りに `exif-js` を使用。いずれも重量ライブラリのため静的 import せず使用時に動的 import する（piexifjs は JPEG 経路のみ・exif-js は EXIF 解析時のみロード。初期バンドルに影響しない。Issue #114）。WebP（RIFF の EXIF チャンク）と PNG（eXIf チャンク）は、取り出した TIFF を合成 JPEG（APP1）に包んで exif-js に読ませる（exif-js は先頭が JPEG SOI でないと読めないため）
- EXIF の書き込みは JPEG（piexifjs）に加え、PNG は eXIf チャンク、WebP は VP8X + EXIF チャンクへ挿入する。バイナリ操作は Canvas / ブラウザ API 非依存の純粋関数 `exifBinary.ts`（単体テスト対象）に切り出し、ブラウザ側の読み取り〜書き込みの橋渡しを `exifTransfer.ts` が担う（piexifjs の動的 import を含むため `readExifTiffFromDataUrl` / `normalizeExifForBakedImage` は async。呼び出し元は await する）。AVIF はメタデータ書き込み非対応

- 言語設定は localStorage に保存

Iteration Guide」のテーマカラー同期 3 点セット参照）。`_headers` で `/sw.js` を `no-cache` にしデプロイごとに新しい SW を確実に取得させる（public/ 配下は `out/` にコピーされて配信される）
- Web Share Target（スマホ共有シートからの画像受け取り、Issue #105）は「SW intercept → Cache Storage → 受け口ページ」の三段構成。静的エクスポートは manifest `share_target` の POST をサーバーで受けられないため、SW（`scripts/sw-template.js`）が fetch handler 先頭で `/share-target` への POST を intercept し、multipart ボディを Content-Type（boundary）ごと一時キャッシュ `wic-share-payload` へ保管してから受け口ページ `/share/` へ 303 リダイレクトする（**保存完了を await してからレスポンスを返す**。`waitUntil` での並行保存は遷移先ページの読み取りが保存前に走るレースがあるため使わない。保管失敗時も受け口へは遷移させ空状態を表示する）。受け口ページ（`src/app/share/`）は mount 時に `readSharedPayload` で読み取り、**エントリをパースより先に削除**してリロード後にファイルを残さない（fail-safe のプライバシー設計。`SHARE_CACHE_NAME` は precache の `wic-precache-<hash>` と衝突しない固定名で、SW の activate では現行バージョン外キャッシュとして削除される = 取り残されたペイロードの掃除を兼ねる。readSharedPayload は読み取りと同時に削除するため、StrictMode の effect 二重実行は ref ガードで 1 回に限定する）。定数・純粋ロジックは `shareTarget.ts` に集約する（`precache.ts` と同じく runtime import を持たず、Node 型ストリップ実行の `generate-sw.ts` から直接 import されてプレースホルダとして SW テンプレートへ注入される）。受け取り後の行き先ツール選択は既存ハンドオフをそのまま使う: 送出 origin は `HandoffOrigin`（`ToolId | "share"`）で、"share" はどの ToolId とも一致しないため送り先候補の自己除外ロジックに自然に乗り、到着バナーは専用文言 `handoff.receivedFromShare` で表示する。manifest の accept（`resolveShareAcceptTypes()`）は受け取り可能ツールの acceptedTypes 和集合とし「受け取ったがどのツールにも渡せない」形式を共有シートに出さない（HEIC / TIFF は convert が受理するため含まれる）。対応環境は Chromium 系 + インストール済み PWA（OS の共有シート統合）のみで、非対応環境の既存動作には影響しない。`/share` は共有シート専用の機能的エンドポイントのため robots noindex + sitemap 除外（`next-sitemap.config.js` の `exclude`）。UI 文言は i18n `share.*`
- ONNX 実行基盤（Issue #100）は ONNX モデル・onnxruntime-web WASM ランタイムアセットの共有ローダー基盤（`modelLoader.ts`）を備える。fetch / Cache Storage 非依存の純粋ロジック（URL 解決・チャンク結合・進捗集計）を単体テスト対象に分離し、WebGPU 優先・WASM フォールバック環境判定は imageUpscaler.ts で実施。ランタイム WASM（26MB）は Cloudflare Pages の 25MiB 上限を超えるため `scripts/copy-ort-assets.ts`（prebuild/predev で実行）で分割配置し、実行時マニフェストに従い取得・結合する（`resolveOrtWasmPartUrls` / `concatChunks` の純粋ロジックで URL 生成とサイズ検証）。モデル（ONNX）と WASM は初回実行時に fetch で取得し、Cache Storage の固定名キャッシュ `wic-model-cache` へ保存（デプロイをまたいで保持される = precache 削除対象（`wic-precache-*`）から除外。ビルド時に `scripts/sw-template.js` の activate で削除スコープを限定）。セッション生成（WebGPU/WASM 自動フォールバック）は Issue #99 で共用化され、`onnxSession.ts`（Canvas / fetch 非依存の純粋ロジック `createOnnxSession`・`buildSessionOptions` 等）が upscale / remove-bg で共有する単一の真実として機能。AI 超解像（`/upscale`）はタイル分割・フェザー（重み付き）合成による大画像対応を備え、タイル演算・テンソル変換は Canvas 非依存の純粋ロジック `upscaleCore.ts`（単体テスト対象）に集約。バッチ処理は専用 Worker `upscale.worker.ts` で 1 ファイルずつ逐次実行（推論自体が GPU / CPU を飽和させるためファイル並列はしない）し、キャンセル（タイル境界の協調停止 + terminate。`upscaleRunner.ts` の `UpscaleBatchHandle`）を提供する。最大入力寸法は 4096px（長辺）に制限し、WASM フォールバック環境でも応答性を保つ（タイルサイズ 192px = 出力 768px）。背景除去（`/remove-bg`、Issue #99）は U²-Net セグメンテーションモデル（約 4.6MB）を ONNX Runtime で推論し、サリエンシーマップを画像の alpha チャンネルとして合成（正規化・バイリニア拡大・フレームバッファ合成は Canvas 非依存の `removeBgCore.ts` 純粋ロジック）。最大入力寸法は 8192px（長辺）に対応。バッチ処理は専用 Worker `removeBg.worker.ts` で 1 ファイルずつ逐次実行（upscale 同様に GPU / WASM 飽和のためファイル並列はしない）。キャンセルは `removeBgRunner.ts` で提供。出力は透過 PNG（既定）または WebP で `_nobg` サフィックスを自動付与
- セキュリティヘッダー（Issue #111）は「グローバル静的定義 + postbuild のページ別 CSP 生成」の二層構成。全パス共通のヘッダー（X-Frame-Options: DENY / X-Content-Type-Options: nosniff / Referrer-Policy: strict-origin-when-cross-origin / Permissions-Policy（camera・microphone・geolocation・payment・usb 無効）/ Strict-Transport-Security（max-age=63072000; includeSubDomains; preload））は `public/_headers` の `/*` ブロックに静的定義する。Content-Security-Policy のみ postbuild（`scripts/generate-headers.ts`）が `out/` の全 HTML からインラインスクリプト（テーマ初期化 + Next.js の RSC ペイロード）の sha256 ハッシュを算出し、`out/_headers` の**マーカー区間**（`# --- generated:security-headers:start/end ---`。複数回実行しても差し替えるだけの冪等マージ）へ**ページ別ルール**として追記する。静的エクスポートでは nonce が使えないためハッシュベース CSP を採用し、script-src は `'self'` `'wasm-unsafe-eval'`（@jsquash 等の WASM 用）+ ハッシュのみで `'unsafe-inline'` / `'unsafe-eval'` は含めない。CSP をページ別ルールにする理由は 2 つ: (1) 全ハッシュを `/*` に集約すると Cloudflare Pages の 1 行 2000 文字制限を超過するリスクがある、(2) Cloudflare Pages はマッチした複数ルールの同名ヘッダーを**カンマ結合する**ため、`/*` にも CSP を置くと二重適用になる（CSP は生成ルールのみに置く）。ハッシュはビルド成果物から毎回算出するため、Next.js 更新やテーマスクリプト変更への手動追従は不要。Cloudflare Pages の `_headers` 制限（1 行 2000 文字・100 ルール。超過分は黙って無視される）は `validateHeadersContent` でビルド時に検査し、超過・HTML 0 件・`out/_headers` 欠如ではビルドを非ゼロ終了させる。純粋ロジックは `securityHeaders.ts`（単体テスト対象。Node API 非依存）に分離し、sha256 算出（node:crypto）と URL 変換（`precache.ts` の `toCacheUrl`）は scripts 側で行う。style-src の `'unsafe-inline'` は style 属性による動的スタイル用（Firefox が style-src-attr 未対応のため style-src 側で許可）。COOP / COEP は SharedArrayBuffer 不使用のため現状不要（将来の WASM マルチスレッド化時に導入）。既知の制限: Cloudflare の 404 フォールバック（任意の不明パスへの 404.html 配信）はページ別 CSP ルールにマッチしないため CSP なしで配信される（`/*` の他ヘッダーは適用される）。運用上の注意: 本番ドメインを Cloudflare のプロキシ（ゾーン）経由にする場合、Rocket Loader / Email Obfuscation / Web Analytics 等の**スクリプト注入系ゾーン機能はハッシュベース CSP にブロックされる**ため有効化しないこと（必要になったら script-src / connect-src への許可追加とセットで行う）。E2E（`e2e/security-headers.spec.ts`）は webServer（serve out）が `_headers` を解釈しないため、postbuild が生成した実 CSP を `page.route` でドキュメントレスポンスへ注入して CSP 強制下の全ページ動作を検証する（SW が HTML をキャッシュ配信すると route を通らないため `serviceWorkers: "block"` を使用）
