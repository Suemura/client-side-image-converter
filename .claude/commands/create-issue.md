要望・アイデア・不具合報告を整理し、GitHub Issue として作成してください。

入力: $ARGUMENTS

このコマンドは `/start-issue` と対になる入口である。曖昧な要望を「そのまま `/start-issue` に渡せる粒度」まで整えてから Issue 化する。

## 手順

### 1. 要件の整理

- 入力（引数・会話の文脈）から「何を・なぜ」を把握する。以下が不明なら**まとめて 1 回だけ**ユーザーに質問する（何度も聞き返さない）:
  - 目的 / 解決したい課題
  - 対象ページ・機能（/convert, /crop, /edit, /redact, /upscale, /remove-bg, /metadata, 横断）
  - 不具合の場合: 再現手順・期待動作・実際の動作・確認環境
- 実現方式に関わる場合は関連コード（`docs/ARCHITECTURE.md` / 該当 `src/` 配下）を軽く確認し、仕様案が既存設計と矛盾しないようにする。**このコマンドでは実装しない**（調査は Issue 本文の精度を上げる範囲まで）
- 大きすぎる要望は分割を提案する（1 Issue = 1 PR で完結する粒度が目安）

### 2. 重複チェック

```bash
gh issue list --state open --search "{キーワード}" --json number,title
```

類似 Issue があれば新規作成せず、既存への追記（コメント）を提案する。

### 3. Issue 本文の作成

以下の構成で日本語で書く（`/start-issue` の planner がそのまま読める粒度）:

- **背景 / 目的**: なぜ必要か、誰のどんな課題か
- **仕様案**: UI・挙動・対応形式・制限値。既存機能との関係（ツール連携・適用範囲・i18n 等）に触れる
- **受け入れ条件**: チェックボックス形式。検証可能な形で書く（単体テスト / E2E の観点を含める）
- **対象外**: 今回やらないことを明記（スコープ膨張防止）
- 不具合の場合は「再現手順 / 期待動作 / 実際の動作 / 環境」を先頭に置く

本文は**リポジトリ外の一時ファイル**（scratchpad）に書き出す（作業ツリーに残さない）。

### 4. ラベルと作成

- `gh label list` で既存ラベルを確認し、`enhancement` / `bug` / `tech-debt` / `documentation` から選ぶ（`/start-issue` のブランチ prefix 決定に使われる）
- 作成前に**タイトル・ラベル・本文をユーザーに提示して確認を取る**（Issue 作成は外部公開操作のため）
- 承認後:

```bash
gh issue create --title "{タイトル}" --label "{ラベル}" --body-file {一時ファイル}
```

### 5. 報告

- Issue 番号・URL・タイトル
- そのまま着手する場合は `/start-issue {番号}` で開始できる旨

## 複数 Issue の一括作成

要望が複数件に分かれる場合は、全件分のタイトル・概要一覧を先に提示して一括で承認を取り、順に作成する（1 件ずつ確認を挟まない）。

## 中断条件

- 目的が不明で、質問への回答が得られない
- 既存 Issue と重複している（追記を提案）
- ユーザーが本文案を承認しない（修正して再提示）
