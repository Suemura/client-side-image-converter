動作確認フィードバックを受けて、該当ブランチ上で修正・検証・push までを行ってください。

入力: $ARGUMENTS

このコマンドは「人間が動作確認 → 気になった点を伝える → 該当 worktree で修正して push」の定型フローである。引数は `{Issue/PR 番号} {フィードバック内容}` の形式（番号省略時は文脈・直近の PR から推定する）。

## 手順

### 1. 対象の特定

- 番号から対象 PR / ブランチを特定する（`/land` と同じ解決順: `gh pr view` → Issue 番号として `*issue-{番号}-*` ブランチを探す）
- 対応する worktree（`.claude/worktrees/issue-{番号}/`）があれば `EnterWorktree` の `path` 指定で入る。なければメイン checkout でブランチに switch する（未コミット変更があれば中断して確認）
- 対象がマージ済み PR の場合: main 上の不具合として扱い、新しいブランチ（`fix/...`）と worktree を切って作業し、最後に PR を作成する

### 2. フィードバックの分析

- フィードバックを項目に分解し、それぞれ「不具合 / 挙動変更要望 / 質問」に分類する
- 各項目について原因をコードから特定する。エラーメッセージが含まれる場合は該当箇所を grep で突き止める
- **原因特定を待たずにユーザーへ聞き返さない**（`workflow-orchestration.md`「自律的なバグ修正」）。再現に必要な情報がどうしても不足する場合のみ確認する
- 修正方針が仕様変更を伴う（Issue のスコープを超える）項目は、修正せず方針を提示して確認する

### 3. 修正と検証

- 項目ごとに修正し、意味のある単位でコミットする（メッセージは日本語、`fix: ...（#Issue番号 動作確認）` の流儀）
- `src/utils/` の純粋ロジックを直した場合は単体テストを追加・更新する
- 検証:

```bash
npm run lint
npm run typecheck
npm run test
```

3 つすべて成功が完了条件。修正内容が E2E でしか検証できない場合（実描画・実エンコード）は、該当 spec があれば `npm run e2e` の対象実行も検討する

### 4. push と報告

- 既存 PR ブランチなら `git push`（PR に自動反映。新規レビューフローは起動しない）
- 新規ブランチの場合は `/start-issue` の手順 10 に従い push + `gh pr create`（単独実行）
- 報告フォーマット:
  - 項目ごとに「原因 → 修正内容 → 検証方法」を 1-3 行で
  - 修正しなかった項目とその理由（仕様判断待ち等）
  - 再度の動作確認をお願いする観点（どの画面で何を試すか）

## 中断条件

- 対象 PR / ブランチを特定できない
- 作業場所に未コミット変更がある
- 修正が Issue スコープ外の仕様変更を伴う（方針を提示して確認）
- 原因特定に必要な再現情報が不足している（不足点を具体的に列挙して確認）
