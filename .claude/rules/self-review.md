# 完了前のドキュメント同期と独立レビュー

## 原則

タスクを「完了」としてユーザーに提示する前に、**独立したコンテキストでレビューを受ける**。自分自身でレビューしてはならない。独立レビューの担い手はタスクの種類で分かれる（下記「レビューの二本立て」）。

## なぜ独立コンテキストか

- 同じコンテキストでの自己レビューはアンカリング・確認バイアスにより甘くなる
- 独立したコンテキストのレビュアーは客観的に評価できる
- メインのコンテキストウィンドウを消費しない

## レビューの二本立て

### PR を作成するタスク（/start-issue 等）

- 独立レビューは **PR 作成後の自動レビューフロー**（pr-reviewer によるインラインレビュー → pr-comment-resolver による対応）が担う。**PR 前の reviewer エージェント起動は不要**（同じ差分の二重フルレビューを避ける）
- PR 前の必須検証: lint / typecheck / test の 3 点成功 + Sprint Contract の自己チェック
- CI（check / e2e）とブランチ保護が最終防衛線として機能する

### PR を作らないタスク

1. 変更が下記ホワイトリストに該当する場合は、先に **docs-sync エージェント**を起動してドキュメントを同期する
2. **reviewer エージェント**を起動し、変更の概要（何を実装したか、どのファイルを変更したか）・差分（`git diff --stat` 等）・Sprint Contract を伝える
3. reviewer のレポート（Pass/Fail + 問題リスト）を確認する
4. **Fail が 1 つでもある場合**: 問題を修正し、再度 reviewer を起動する（ループは 3 回まで。超えたら未解決の指摘とともにユーザーへ報告）
5. **全項目 Pass**: ユーザーに提示してよい

## docs-sync の起動条件（ホワイトリスト・single source of truth）

以下の**いずれかに該当する場合のみ** docs-sync を起動する:

1. **ユーザー向け機能の追加・変更**: 新ページ・UI 挙動・対応形式・制限値の変更（README 日英の更新が必要になるもの）
2. **開発コマンド・ビルド / CI / デプロイ構成の変更**: package.json scripts・GitHub Actions・デプロイ設定
3. **ハーネスの変更**: `.claude/` 配下（agents / commands / rules / hooks / settings）・ワークフロー
4. **ディレクトリ構造・パスエイリアス・テスト方針の変更**

**起動しない**（明示）: バグ修正・リファクタ・内部実装変更・依存更新・テスト追加のみの変更。`docs/HISTORY.md` の変更ログは**メインエージェントが直接 1 エントリ追記する**（docs-sync の起動理由にしない）。

## サブエージェントへのコンテキスト手渡し（探索削減）

docs-sync / reviewer を起動する際は、プロンプトに以下を必ず埋め込む:

- `git diff origin/main...HEAD --stat` の出力（未コミット変更があれば `git status --short` も）
- 変更概要（何を・なぜ、2-3 文)
- （reviewer のみ）Sprint Contract

エージェントは渡された差分から読み始め、コードベース全体を探索しない（各エージェント定義の「探索削減」参照）。

## 適用タイミング

- コード変更を伴うタスクの完了時
- 複数ファイルにまたがる変更の完了時
- ※単一ファイルの軽微な修正（typo 等）は対象外（docs-sync も同様にスキップしてよい）
